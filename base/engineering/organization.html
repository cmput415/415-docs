CMPUT 415 - Compiler Design

Software-Engineering Process

The organization of the teams, the creation of a design document early on, the establishment of methodology to write code, review code, enforce code style, all will be very important for the successful completion of the project. This page sets up expectations and minimum requirements that each group must follow.

Project-Review Meetings

Minimum Frequency: once a week
Duration: at most one hour
Meeting Notes: required
Attendance: required and recorded
Discussion items/agenda: circulated prior to meeting
Actions and outstanding issues: recorded prior to the end of the meeting
Note Recording: Use the same file to record notes of all meetings. Add a new meeting header to the end of the file to start recording the notes for each meeting.
Notes file: ProjectReviewMeetingNotes in project repo

The goal of project-review meeting is to discuss all aspects related with the project.

In each project-review meeting the individual progress for each member of the team must be reported and recorded in the notes. Outstanding issues must be discussed and can be removed from the list of outstanding issues. New issues might be added to the list. Discussion items must be recorded in the meeting notes. Actions must be recorded. Attendance records must indicate what time individuals that came late or left earlier arrived or departed.

First Project Meeting

The first of such meetings is an organizational meeting. This meeting may be longer than the project-review meetings, but should not last more than two hours. It may also be split into two meetings on the same week. At a minimum by the end of the first meeting the group must have agreed on the following:
  • a mechanism to create the project design;
  • code style and tools to support enforcement of code style (see resources below);
  • a process to create pull requests and for code review. You can choose the workflow, but it must be documented so that it is followed. We suggest the following:
    • Create Branches for features.
    • Create pull requests (PRs) to merge back.
    • Add at least one reviewer for each PR.
  • a conflict resolution mechanism --- majority vote? a team leader decides?
  • testing strategy --- or a process to create one
  • who will chair the meetings, who will write the notes, who will be the scrum leader;
  • time for all the project meetings -- designate someone that will create calendar entries;
  • an initial list of outstanding issues.
  • the initial list of actions --- every action must have at least one team member responsible for it.

Scrum Meetings

Minimum Frequency: twice a week -- higher frequency recommended
Duration: at most 15 minutes
Meeting Notes: brief, required
Notes file: ScrumMeetingNotes in project repo

Geekbot has some interesting advice on Scrum meetings.

Check-in Meetings

Frequency: as needed
Attendance: as needed
Meeting Notes: not required
Goals: report blockers, request help, discuss issues
